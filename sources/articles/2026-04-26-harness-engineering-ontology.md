# Harness Engineering 的语义底座：本体驱动的 Agent 可控执行

> 周士雄 @ DataFunSummit

## 核心命题

Agent 进入真实业务后问题高度相似：术语用不准、推理逻辑跑偏、执行结果跟企业规则对不上，在关键节点**「自信地做错事」**。

根本原因：Agent 缺少一个「懂规矩的结构」——它知道怎么干活，但不知道行业边界、企业决策规则在哪里。

## 方法论三支柱

1. **架构约束** — 限定行为边界
2. **上下文工程** — 管理信息输入
3. **反馈闭环** — 验证输出质量

## 本体驱动方案

用 Ontology（本体）为 Harness 提供语义基础设施，使约束不再来自工程配置，而直接来自**业务结构本身**。

## 架构三层

| 层级 | 职责 |
|---|---|
| 本体层（知识底座） | LPG 标签属性图，定义 Entity/Relation/Event/Action/Logic |
| 认知引擎层 | 翻译 + 仲裁：启动前注入知识，执行后约束校验 |
| Agent 执行层 | 纯粹执行者，工具/边界由本体决定 |

## 反馈闭环

- **客观校验**替代主观评估：硬约束直接规则验证，不依赖模型理解
- **本体持续进化**：Agent 执行数据反哺本体，识别未覆盖概念和频繁出错位置
- **可解释性**：每次校验有明确结构依据，可追溯到具体规则节点

## 五大元模式

1. **Entity** — 业务对象（工单、设备、人员）
2. **Relation** — 实体间的语义连接
3. **Event** — 业务中有意义的状态变化
4. **Action** — 可执行操作，含触发条件和参数约束
5. **Logic** — DAG 编排引擎定义的执行流程

## 核心洞察

> 企业 AI 的竞争，最终不是模型之间的竞争，而是谁先把自己的业务知识结构化。
> 真正的护城河，从来不是买来的工具，而是沉淀下来的认知。
